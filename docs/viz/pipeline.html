Build Pipeline & Orchestrator

Nxcleus turns a sanitized brief into a certified, tested, deployed process — event by event.

A single event-sourced asyncio engine drives every job through stages 0→7. Each stage is an asyncio task given five capabilities — emit, complete, dao, meter, checkpoint. The engine is deterministic: it dispatches, retries, and resumes, but it never decides. Deciding lives in the certified plan and in the conductor's bounded, hash-chained wave reviews. Every state change is appended to one events table and fanned out over SSE; the UI is a fold of that log, the DB rows are the materialized present.

Engine Mechanics

One deterministic driver, one append-only log

The engine holds a task per job and per run. _drive_job loops: read the job, act on its status, run the stage, repeat — stopping only on a parked or terminal status. It decides nothing about content; it only advances the machine.

The state machine

Status is written to jobs.status and emitted as job.stage_changed in the same call, so persisted state and the event log can't diverge. test_state_machine.py asserts every declared stage appears in order and that exactly one process registers per job.

Statuses from engine.py — _STAGE_MODULES + _STOP = {quoted, blocked, done, aborted}. quoted parks until approve-quote (sandbox auto-approves).

Event sourcing & replay

emit(scope, type, payload) writes one append-only events row (monotonic seq) and fans out to every live SSE subscriber. UI state for a job is a fold of its events; the DB rows are the materialized present.

  • Token deltas (plan.delta, task.output_delta) are coalesced to ≤10/s per scope
  • replay(scope, from_seq) and replay_prefix(type_prefix, …) reconstruct any view
  • Named SSE events; slow subscribers are dropped and reconnect with from_seq

Resumability & idempotency

On boot, engine.start() scans resume_candidates() (non-terminal jobs) and resume_runs(), resubmitting each. A crash mid-stage-4 resumes at the top of the stage.

  • Deterministic IDs (deterministic(kind, …seed)) make writes upserts, not duplicates
  • Done build_tasks are skipped on the retry — no re-generation
  • test_resumability.py: rewind to building, drop the process, re-run → exactly one process, N tasks, no dupes

Error, retry & budget policy

Each stage runs under a whole-stage watchdog and a uniform failure policy in _run_stage.

watchdog
per-status timeouts — planning 600s, certifying 900s, building 1800s, consolidating 480s, qa 1200s, intake 300s, delivering 120s
retry
_MAX_STAGE_ATTEMPTS = 2: first failure emits a warn notice and retries once; second failure → blocked
budget
BudgetExceeded is a deterministic guard, not a fault — no retry; the job → aborted gracefully (partial dashboard survives)
cancel
CancelledError re-raises untouched so engine.stop() can tear tasks down cleanly
Wave Orchestration

Topological waves, then a conductor between each

Stage 4 turns the plan DAG into ordered waves, dispatches each wave concurrently to the coder pool with capability routing, and lets the conductor amend the road ahead — never the road already built.

Waves from the DAG — Kahn levels

waves_from_dag(dag, modules) computes topological levels: every task whose deps are already done forms the next wave. A dependency cycle is a guard, not a crash — the remainder is emitted as one wave. With no DAG it falls back to a single wave of all modules.

Per wave: parallel _build_task under a semaphore, then review_wave (only when total_waves>1 or BoM's conductor.always).

Capability routing — pool.pick_member

Deterministic argmax over task_flags. Scores: strong=+2, ok=+1, weak=−2. Ties break to least-loaded, then round-robin. No positive score → round-robin so routing never blocks. The decision (chosen, score, considered, reason) rides in the task.started payload — the BoM panel shows not just which model built a module but why.

Proceed-without-review (D8 / 07 §3.1)

A conductor that raises or times out never dead-ends the build — it emits a warn notice and a green_flag {reviewed:false}, and the wave proceeds. An amendment aimed at an already-built region is rejected by scope-lock; a malformed patch is skipped rather than corrupting the live plan.

Amendments & Consults

Constrained patches on a hash chain, sanitized escalations through the gate

Both the certifier and the conductor change plans the same disciplined way: an RFC-6902-style patch, scoped to named regions, appended to a tamper-evident chain. When a fix needs the frontier's help, it goes back through the egress gate as a sanitized consult.

The amendment hash chain

Every amendment links to its predecessor. The genesis is 64 zeros; each hash is sha256(prev_hash + patch_json + rationale). Because the plan-ref, check, and origin (certifier | conductor) are recorded, the full amendment history in the delivered package is auditable and re-derivable.

One consult round from certify/stage2.py:_run_consult. Everything planner-bound — the question and each finding — passes trust.sanitize_consult first.

Region vocabulary

_merge_regions replaces only the named regions — modules, interfaces, dag-tasks, topology steps — resolved through region_ids / normalize_regions. Non-named regions are untouched.

Consult cap

_CONSULT_CAP = 3. Past it, remaining structural findings pause the job for human review; non-structural ones are dropped. A failed consult (scope-lock violation, timeout, refusal) degrades gracefully — it never blocks the build.

Conductor parity

The conductor uses the exact same mechanics between waves — apply_rfc6902 on the live plan dict, then append_amendment with origin="conductor" and the prev_hash chained.

Stage 6 · Adversarial QA

Probe a live endpoint, recompute blind, judge against the goal

QA doesn't grade a mock. It deploys the assembled process to a real HTTP surface, sends an inspector swarm at it, and runs a Numeric Oracle that recomputes each answer independently — then compares the two. Disagreements are flagged, never silently trusted.

Probes and the oracle's "actual" both hit the same live shim; it's torn down after both finish. From qa/stage6.py:run.

The Numeric Oracle protocol (dual implementation)

  • Expected: oracle.compute recomputes from the sanitized rule text alone, self-consistency k=3 — never from the plan's code
  • Actual: _staged_actual POSTs the vector inputs to the deployed process and reads risk_score/score/… from the real response
  • Verdict: match / mismatch / no_actual / oracle_uncertain, compared under the vector's tolerance
  • No obtainable actual records no_actual honestly — a match is never invented
  • mismatch or oracle_uncertain → a disagreement ticket for a human

Ticket lifecycle

Findings become tickets and move through a bounded loop (_FIX_CAP = 3):

  • open → in_fix → verified when the coder resolves it
  • A fix that can't be applied → human_review
  • Oracle disagreement tickets skip the fixer and go straight to human_review
  • Goal unfulfilled with a blocker gap → ticket + RuntimeError → job parks
◆ The goal-fulfillment check gets evidence, not a bare goal

The final judge receives a manifest of what was actually delivered — topology steps mapped to goal clauses (screen→sanctions, score→risk, decide→route), integration tests passed, oracle vectors recomputed, scenarios probed, open defect count — so it judges fulfilment against real evidence rather than defaulting to "unfulfilled". An infra timeout on this check degrades to partial with a caveat; it never hard-blocks on a network hiccup.

Code-Exec Sandbox

Generated code runs untrusted, network off, write-isolated

Coder output is untrusted, so it executes in a throwaway python:3.12-slim container with no network and hard resource caps. The job workspace is mounted read-only and copied into a container-private tmpfs — the sandbox can never write back to the control plane.

What runs

run_tests runs the workspace's own pytest suite when the coder generated one; otherwise a per-file py_compile import/compile smoke of every .py — proving the generated code loads and runs in isolation.

Limits & exits

120–180s timeouts; return codes are meaningful — 124 timeout, 127 no docker, 126 daemon error. A missing daemon or image never crashes a stage.

Graceful degrade

No Docker (dev box, CI) → the Wave-1 deterministic behaviour, flagged sandboxed:false so the UI badge shows it wasn't truly sandboxed. Mock mode and tests never hard-depend on a daemon.

▲ compileall is unreliable — the code works around it

The smoke test gates on a per-file py_compile (which returns non-zero on syntax errors) rather than compileall (which returns 0 even on syntax errors). The exit code stays 2 when the workspace has no .py at all, so an empty or unshared mount is reported as a failure, not a false pass.

Routing & the Data Boundary

The one place a dispatch happens — and the wall is code, not policy

Every seat call goes through router.complete. It resolves the seat, applies the sovereign overlay, checks fleet health, enforces the zone × data-class matrix (fail-closed), guards the budget, dispatches, and validates structured output with one repair round — logging a meter row, an egress row, and a model.call event per dispatch.

Honesty Strip

Spec vs Code · what's real vs what's stubbed

The pipeline runs end-to-end today, but some stages carry real machinery while others are Wave-1 skeletons that keep the walking-skeleton coherent. Here's the honest divide, straight from the code comments.

Where the code diverges from the specs

▲ Spec 03 · Stage 3 is not a driver stage

The specs describe Quote as its own pipeline stage. In code there is no quoting status in _STAGE_MODULES — stage3.issue() is a deterministic sub-step called at the tail of Stage 2, after which the job parks at the quoted stop-state until approve-quote.

▲ Spec 07 · duplicated transition helper

Engine.set_stage and StageContext.advance are byte-for-byte the same operation (write jobs.status + emit job.stage_changed). Stages call ctx.advance; the engine's own transitions call set_stage. Harmless, but two names for one truth.

▲ Spec 07 §3.1 · review is conditional

The conductor only reviews when total_waves > 1 or the BoM sets conductor.always. A single-wave build proceeds unreviewed by default — and the conductor seat has no fallback binding, so if node B is down the review is skipped entirely (covered by proceed-without-review).

Real & proven

  • Boundary enforcement — router._check_boundary is fail-closed; RAW→EXTERNAL raises, logged to the egress ledger. A live 0→7 E2E ran at $1.26 (commit log)
  • PII mask → vault → sanitized brief — the mechanical boundary in Stage 0 is real; raw values never serialize outward
  • Live staging + dual-implementation oracle — staging.deploy stands up real HTTP; the oracle compares blind recomputation against a real POST /run_unit
  • Docker code-exec sandbox — --network none, read-only mount, tmpfs; real pytest / py_compile when Docker is present
  • Event sourcing + replay — append-only events, delta coalescing, replay(from_seq)
  • Resumability — deterministic IDs + build_task_done skip; asserted by test_resumability.py
  • Topological waves, scope-locked amendments, hash chain — Kahn levels, RFC-6902 patches, sha256 chain, consult sanitization gate
  • Per-call metering — real cost/egress/model.call rows per dispatch, quote↔invoice reconciliation

Stubbed / Wave-1 skeleton

  • Intake dialogue — single-shot auto-confirm; the multi-turn clarification + human confirm-spec gate is Wave 2
  • Placeholder seats — when no real seat / model is live, _placeholder.py drives MockClient from fixtures; the conductor placeholder always amends an early-wave region, the oracle marks vectors ending in "2" uncertain (to exercise the beats)
  • Operate drive_run — units go through seats directly, not a process-runtime container (Wave 2); per-unit status is synthetic (i%7, i%11); sampling is a TODO(wave2) constant; one spot-check discrepancy is a rehearsed warranty beat
  • Consolidation fix loop — a skeleton; run_tests returns green so it rarely loops
  • Process fan-out with no corpus — falls back to nominal unit refs (marked ponytail:); real customer corpus intake is post-deadline
  • Refine — records a compact diff (tests_added: 0) rather than a scoped rebuild
