Control-plane reference · specs 06

Nxcleus — API,
Events & Routing

The complete request surface of the Nxcleus control plane: a FastAPI app whose orchestrator runs as asyncio tasks in one process, a single-writer SQLite event log that fans out over Server-Sent Events, and one router where every model dispatch is boundary-checked in code. Everything below is read straight from the source — the spec is secondary, and where code diverges it is called out.

9 routers + 3 app routes 40+ REST endpoints 4 SSE channels 90+ event types 1 dispatch chokepoint FastAPI · SQLite · React/EventSource
01 · The shape of it

One process, one writer, one boundary

The whole control plane is assembled in create_app() (backend/app/main.py). Nine routers mount under /api; the orchestrator engine and a fleet poll-loop start inside the lifespan and run as asyncio tasks — no IPC, no worker pool. UI state for any job is a fold of its events; the SQLite rows are the materialized current state.

02 · Who may write

A deliberately hackathon-grade auth model

All reads and all SSE streams are public. Writes split four ways, defined in backend/app/api/deps.py. The pivot is one env var: ADMIN_TOKEN. When it is empty (dev / open demo) every demo and admin write is open; the lifespan prints a loud warning in non-mock mode so an unguarded demo can't sneak past unnoticed.

TierDependencyCredentialGuards
public read— none —none all GETs, all SSE streams, /replay
demo writerequire_demo_token X-Demo-Token == ADMIN_TOKEN job create/mutate, quote approve, runs, refine, instantiate, connections, seat binds, sovereign toggle
admin / noderequire_admin_token X-Admin-Token == ADMIN_TOKEN node self-register, node drain, sovereign set
sandboxsandbox_session sandbox_session cookie (auto-set, httponly) judge-sandbox runs — anonymous, rate-limited 3/hr per session
processverify_token() X-Process-Token (HMAC) the model-proxy — deployed process code's only egress (see §7)
Note Empty token = open

require_demo_token and require_admin_token both short-circuit to allow when settings.admin_token is falsy. The same token backs both the demo and admin tiers — there is no separate admin secret. The frontend attaches X-Demo-Token from localStorage on any request flagged demo:true (api/client.ts → req()).

04 · The event system

One append-only log, four ways to read it

backend/app/events.py is the whole bus. emit(scope, type, payload) writes one row into the events table (RETURNING seq — the global sequence number doubles as the SSE id:) and fans out to every live subscriber whose scope or type_prefix matches. There is exactly one writer.

api/sse.py · _format_event() ~L17 — the wire envelope
# NAMED event: `event: <type>` — so the browser needs addEventListener PER type
return f"id: {event['seq']}\n" \
       f"event: {event['type']}\n" \
       f"data: {json.dumps({seq, ts, scope, type, payload})}\n\n"

The four subscription shapes

ChannelEndpointSubscribe byCarries
job:{id}/jobs/{id}/eventsscope matchthe entire stage-0→7 pipeline for one job (also sandbox + refine jobs)
run:{id}/runs/{id}/eventsscope matchrun.*, spotchecks, warranty tickets, review.decided
fleet/fleet/telemetryscope matchtelemetry.gpu (per-GPU), fleet.node_ready/down
egress.*/egress/streamtype prefixevery egress crossing across all scopes — the network monitor
Mechanics Replay-then-tail, heartbeat, resume

Replay-then-tail: sse_response() subscribes before replaying so no event slips through the gap, sends the backlog from from_seq, emits a : connected comment, then tails. Heartbeat: a : heartbeat comment every sse_heartbeat_s (10 s) when idle — invisible to EventSource, so a completed job's stream just stays "open". Resume: Last-Event-ID (the last seq) takes precedence over ?from_seq; browser reconnects resume automatically. Backpressure: a full subscriber queue (1000) drops the event — the client catches up on reconnect via from_seq. Coalescing: plan.delta and task.output_delta are token-level; they are buffered per (scope, type, key) and flushed at most sse_throttle_per_s (10/s), so tokens stream without flooding the log.

Seam The system scope has no reader

config.* and admin system.notice events are written to scope system, but no SSE endpoint subscribes to it — every stream is job:, run:, fleet, or the egress. prefix. Those events land in the log (readable via /replay/system) but never reach a live UI stream.

The frontend seam — named events + a tolerant normalizer

Because the backend sends named SSE events, plain onmessage never fires. frontend/src/api/sse.ts registers the same handleFrame handler once per known type, then hands each raw frame to normalizeEvent() (api/adapt.ts), which conforms ~30 diverged backend shapes back to the typed NxEvent catalog the UI folds over.

frontend/src/api/sse.ts · openEventStream() ~L54 — one listener per type
// Named events don't fire onmessage; register the handler per type.
es.onmessage = handleFrame;                          // unnamed fallback
for (const type of KNOWN_EVENT_TYPES)
  es.addEventListener(type, handleFrame as EventListener);

// handleFrame: parse -> track lastSeq -> normalizeEvent(raw) -> onEvent
const raw = JSON.parse(e.data);
if (typeof raw.seq === "number") lastSeq = Math.max(lastSeq, raw.seq);
const ev = normalizeEvent(raw);   // backend payload -> typed NxEvent; null => dropped
if (ev) handlers.onEvent(ev);
frontend/src/api/adapt.ts · normalizeEvent() ~L181 — tolerant per-field fallback
case "boundary.sanitized": {
  // prefer a typed field if present, else DERIVE it from the backend's raw counts
  const pii = num(p.pii_fields_masked) ?? 0;
  const findings = arr(p.findings).length ? p.findings
    : [ pii > 0 && { rule_id: rules[0], label: "PII fields masked", count: pii } ]
        .filter(Boolean);
  return env(raw, "boundary.sanitized", { findings, never_leaves, brief_tokens });
}
Why it matters The adapter is the real contract

The normalizer's own header admits it: "spec 06 §3 is the intended contract; the live backend diverged across ~30 types, so we conform the frontend to reality here." Every UI shape is defined by what normalizeEvent emits, not by the spec — and any type it doesn't case on returns null and is silently dropped before the fold. See the divergence list in §8.

06 · Model-call routing

The one place a dispatch happens

models/router.py → Router.complete() is the single chokepoint. Every internal stage call and every /proxy/complete call goes through it, and the boundary is enforced here, in code — not in config. 07 · Proxy-token scheme

A self-contained HMAC capability

Generated process code never holds a model key. It calls the control-plane proxy, which dispatches to the process's allowed seats on its behalf. That call is gated by a compact HMAC token scoped to one process id + a seat allowlist — a token minted for process A cannot drive seats for process B. backend/app/boundary/proxy_token.py, no server-side store.

ElementValue
Format<b64url(claims)>.<b64url(hmac_sha256)> — self-contained, stateless
Claims (signed){process, seats:[...], exp} — canonical JSON (sorted keys)
AlgorithmHMAC-SHA256 over the b64url claims body
Signing keysha256("nxcleus:proxy-token:v1:" + ADMIN_TOKEN); a stable dev key when ADMIN_TOKEN is unset
TTLdefault 3600 s; verify_token() rejects expired, never raises
Verificationhmac.compare_digest (constant-time) → exp check → return claims or None
Enforced at/proxy/complete (header X-Process-Token) and the staging shim's POST /run_unit (header x-proxy-token)
Divergence Two header names, and nothing mints tokens in the app

The control-plane proxy reads X-Process-Token; the staging shim reads x-proxy-token — two names for the same capability. And sign_token() is never called from application code — only from tests. In the live path, stage-6 QA deploys the staging shim with expect_token=False, so the wrong-tenant boundary exists in code and is proven by tests, but the running demo doesn't yet mint or require the token. See §8.

extraEndpoints and behaviours beyond the spec surface
Several surfaces exist in code that a 06-only reader wouldn't expect: the whole /proxy/complete egress path, POST /processes/{id}/instantiate (re-enter at stage 4, no planner call), POST /processes/{id}/refine, the type-prefix /egress/stream, voice-policy intake (multipart → Whisper) on /jobs/{id}/policy, and the GET /api/replay/{scope:path} catch-all. noteNo Pydantic request models — bodies are hand-parsed dicts
Every write handler takes body: dict and reads keys with .get(). Validation is manual and permissive (missing keys default rather than 422). The typed contract lives entirely on the response side and in the frontend's client.ts interfaces.
