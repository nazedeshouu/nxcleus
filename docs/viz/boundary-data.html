Nxcleus control plane · reference

Data Boundary,
Persistence & Operations

Nxcleus builds and runs data-processing software for regulated enterprises without their sensitive data ever reaching a frontier model. The premise is mechanical, not a promise: only SANITIZED data crosses to EXTERNAL; raw customer records, the vault that maps them back, and every code-touching model stay in the LOCAL zone. The boundary is enforced in one function, logged on every dispatch, and proven by tests.

The story has two data classes and four zones. Data is born RAW — the customer's real names, accounts, documents. Stage 0 masks it into typed placeholders and stores the reverse map in a boundary vault that never serializes outward. What leaves stage 0 is a SANITIZED brief. Only that brief is allowed to reach the one seat that lives EXTERNAL — the planner, on a frontier model. Everything that actually touches the customer's data — certification, code generation, review, QA, the running process — happens on LOCAL GPUs where RAW is legal.

This page reads the code, not the pitch. Every gate below names the real file and function that implements it; every divergence from specs 01 / 04 / 05 / 09 / 10 is called out where it occurs.

§1 · The boundary

What crosses the line, and through which gate

Four zones, mapped from a model backend's kind in registry.ZONE. The LOCAL wall holds all raw data and every RAW-class seat. Exactly one artifact — the sanitized brief — is designed to cross into EXTERNAL, through exactly one gate.

LOCAL — raw legal AMD_HOSTED — fallback EXTERNAL — sanitized only CUSTOM — attested vault / consult gate blocked crossing

§3 · Intake pipeline · stage 0

How raw becomes a brief

Stage 0 (boundary/intake.py) is the one place RAW data is admitted. The mask, exactly

Six patterns, ordered most-specific first so a card number isn't shadowed by the phone pattern. Placeholders use guillemets («KIND_N») so they can't collide with source text, and are numbered per-kind.

The vault — placeholder ↔ raw

Table boundary_vault(job_id, placeholder, raw_value, kind), PK (job_id, placeholder), upserted so a re-run is idempotent. Three directions of substitution, each for a different phase:

get_map
placeholder → raw (read the whole map)
rehydrate_text
placeholder → raw · LOCAL only (D9)
reverse_substitute
raw → placeholder · longest-first, before a consult crosses
count
vault size → the sensitivity report

The vault never appears in any *_json package column, event payload, or log. It is the one table whose whole purpose is to not leave the box.

Multimodal intake seams

Two local extractors feed the mask, both structured so a future upgrade is a config change not a rewrite:

OCR — boundary/ocr.py

Tries pytesseract (prod VM only), then a co-located .txt sidecar (the PIL-generated KYC docs ship one, so OCR works with zero system deps), then a PDF text layer. Returns (text, method); never raises.

Voice — boundary/whisper.py

Local whisper.cpp on the VM CPU for policy dictation. Recording and transcript never leave the box. Disabled (raises WhisperUnavailable) when WHISPER_MODEL_PATH is unset.

Encrypted secrets — boundary/secrets.py

BYOK keys are write-only: stored as Fernet ciphertext in the secrets table, surfaced masked (abc••••wxyz), never serialized into events, logs, or packages. api_key_ref columns point here; the key material itself is only decrypted at the dispatch boundary. In dev an ephemeral Fernet key is generated (keys don't survive a restart — acceptable for the hackathon).

§4 · Egress & proxy tokens

One outbound client, one ledger, signed access

All outbound HTTP leaves through a single httpx.AsyncClient; every dispatch — real or mock — writes an egress_log row via egress.record(). Generated code never holds a model key: it calls the control-plane proxy with an HMAC token scoped to one process and a seat allowlist.

The egress ledger

egress.record() is called by the router at exactly two points: when a call is blocked (bytes 0, logged before the raise) and when a call is dispatched (bytes ≈ tokens×4). Each row carries scope, host, zone, seat, byte counts, and a sovereign_violation flag. The "network monitor" panel is a SELECT over this table — refusals included.

Per-process proxy tokens

A running process container's only network egress is POST /proxy/complete. The request carries an X-Process-Token: a self-contained HMAC-SHA256 token (b64(claims).b64(mac)) with claims {process, seats[], exp}. The signing key is derived from ADMIN_TOKEN; there is no server-side token store.

Defense in depth at the proxy — api/proxy.py

The proxy checks the token, then re-checks the seat against the manifest's BoM allowlist read from disk, then dispatches through the same router (same boundary + budget enforcement as an internal call). Meter scope is the process id.

§5 · Persistence

The database — one event spine, projected aggregates

SQLite in WAL mode over aiosqlite (SQLAlchemy Core), single writer. The events table is the ordered spine; every other table is a queryable projection the stages write through the DAO. All *_json columns are (de)serialized in one place (db/dao.py) so callers work in dicts.

Event sourcing, honestly

The events table is append-only with a global seq. It's the SSE source and the replay tape. But the system is not pure event-sourcing: aggregates (jobs, plans, runs…) are also mutated in place as projections. The event log is the narrative; the tables are the current state.

Single writer, WAL

The API process is the only writer. An asyncio.Lock serializes writes so concurrent stages can't interleave a transaction; reads run lock-free against the last committed WAL snapshot.

Hash-chained amendments

Every plan amendment is chained: hash = sha256(prev_hash + patch_json + rationale), with a per-plan seq. Certifier and conductor amendments share one chain, so the plan's edit history is tamper-evident.

Spec 05 vs code

Pydantic models ≠ SQL schema. db/models.py defines the artifact shapes that fill the *_json columns (validated against Model.model_json_schema()), roots set extra="allow" for additive evolution. The table schema is db/schema.sql, applied idempotently at startup. The checkpoints table is an explicit additive extension to spec 05 (backs StageContext.checkpoint() for resumability, 07 §1/§4).

§6 · Operate & registry runtime

From a certified job to a running process

Delivery assembles an immutable package on disk, registers a process + process_version v1, and links its first run. Thereafter the operate phase drives batches over that version — every unit judged by a LOCAL seat, zero frontier calls. Stage-6 inspectors probe a real HTTP endpoint, not a mock.

The immutable package layout

Per-job work happens in data/workspaces/{job_id}/src; the delivered package lands at data/packages/{slug}/{version}/. assemble_package() writes a fixed tree:

Register — delivery/stage7.py

Slugifies the title (uniqued), builds the manifest, computes the reconciled invoice, then create_process + create_version(v1). The build-time corpus run is re-linked to the new process id so the registry dashboard fills.

Version & diff

process_versions holds package_path, an image_tag (build mode only — process mode is null), and a diff_json vs the previous version. Superseded versions stay runnable.

Run — runtime/operate.py

drive_run() executes a version over a batch: per-unit judgment through a LOCAL seat (data_class="SANITIZED"), progress + meter ticks, and a run-level cost with frontier_calls: 0.

Staged live deployment — the inspector's real target

At stage 6 the consolidated process is served over real HTTP by a FastAPI shim on an ephemeral 127.0.0.1 port, so the inspector swarm attacks a live surface. It tries to import the assembled process.py's run_unit entrypoint; a broken build is said out loud rather than silently stubbed. /run_unit enforces the per-process proxy token when armed.

§7 · Judge sandbox

Three synthetic companies, one FIFO queue

The judge sandbox lets a visitor run the full pipeline on believable-but-synthetic corpora. A sandbox job is a normal job (origin=sandbox, mode=process, auto-confirmed spec, capped auto-approved quote) — the guards, not the pipeline, are what's special. Synthetic data is SANITIZED by construction, so it can safely appear in the planner brief.

Meridian Bank bank

800 customers · 1,200 accounts (~8% dormant) · 8,000 transactions. Planted: sanctions-adjacent names, ~15 dormant reactivations, 12 structuring runs (deposits just under $10k).

Aurora Clinic clinic

Encounters, vitals, billing. Prompts: duplicate-billing shapes, impossible vitals as data-quality flags, overdue-screening cohorts.

Hale & Ostrom lawfirm

A contracts/*.txt corpus (a ~200-unit sweep animates well). Prompts: renewal dates & auto-renew clauses with sub-60-day notice windows, missing signature blocks, fee-cap breaches.

Seed kits — real lists, planted inputs, live outputs

Datasets are generated from a fixed RNG seed (SEED = 20260709) so a regenerate is byte-for-byte reproducible (rehearsal gate 09 §7 — "seed regenerated from scratch proves no hand-tuned state"). The KYC hero kit ships the real public OFAC SDN + EU consolidated sanctions lists, downloaded at seed time, with a graceful synthetic fallback that still carries the planted hit-names when the network is unavailable.

How a sandbox run becomes a real process job

There is no separate sandbox execution path. The enqueued job runs the standard pipeline; in process mode, run_process_fanout() loads units straight from the company's read-only SQLite corpus, streams each through the plan's per-unit topology steps on the seats it names, and writes them under a real runs row (an FK parent). At delivery that run is linked to the registered process.

Safety guard worth noting

A live planner can invent arbitrary seat names in the topology. Because fan-out runs with data_class="RAW", any step naming planner (or an unknown seat) is silently rewritten to coder — a LOCAL, RAW-ceiling seat — so a hallucinated topology can never route real corpus rows to an EXTERNAL model. The boundary holds even against the plan's own output.

The test_sandbox_path.py regression proves the whole path: the enqueued job reaches done with a real fan-out run, FK-valid units, and a process link — and that FK enforcement holds on every pooled connection (the Wave-2 bug was PRAGMA foreign_keys set on only one).

§8 · Metering & economics

Metering is the product claim

Every dispatch writes a meter_events row with real token counts (from the provider's usage) and a cost from the rate card. Local tokens are priced as GPU-seconds, not per-token. The quote is a range; the invoice is exact — reconciled side by side at delivery. This is the "frontier is capex, the rest is your GPUs" money slide, straight from the ledger.

The cost formula

Two budget guards short-circuit dispatch and raise BudgetExceeded: a Fireworks daily spend cap, and a per-scope cap (the sandbox registers one at $0.50/run).

Spec 10 vs code

The built-in defaults in rates.py (claude-fable-5 $3/$15) are stale placeholders; infra/rates.yaml overrides them with the confirmed $10/$50 when present, and load_rates() deep-merges YAML over defaults. If the YAML is unreadable it prints a warning and bills the built-in card rather than silently — "stale prices silently billing the money slide would be a demo lie."

§9 · Honesty strip

What's real, what's stubbed, where code diverges from spec

The boundary mechanics, enforcement, ledgers, persistence, and metering are real and tested. The higher-level orchestration has deliberate Wave-1 depth limits, each marked in the code. Nothing here is hidden behind a mock that pretends to be live.
