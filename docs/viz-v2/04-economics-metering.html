Economics & Metering

Metering isn't bookkeeping garnish here — it is the product claim. "Frontier intelligence is a capital expense, not a marginal cost" only holds if the receipt is real. So every model dispatch writes a metered row from actual token counts and real rates; stage 3 turns the plan into a deterministic quote (a range); stage 7 aggregates the metered rows into an exact invoice; and the two are reconciled side-by-side at delivery. Below: how a row is written, how local GPU time is attributed honestly, the rate card, the quote engine, the invoice, the money-slide summary, and the budget guards — traced end-to-end through a KYC/AML onboarding build. Ground truth: spec 10, infra/rates.yaml, backend/app/metering/*, quote/stage3.py.

§1

Every dispatch writes a metered row

spec 10 §1 · router.py _record_dispatch() → meter.record()

There is exactly one place cost is born: the router. After every model call — planner, certifier, coder, oracle, a schema-repair retry, a mock in dev — router._record_dispatch() writes a meter_events row, records the egress ledger, emits a model.call event, and throttle-broadcasts a meter.tick. Nothing bills except through this seam, so the invoice can never disagree with what actually ran.

§2

GPU-second attribution — an honest approximation

spec 10 §2 · meter.estimate_gpu_seconds()

Local tokens on the MI300X fleet are not priced per-token (rates.yaml local: 0/0) — self-hosted intelligence is GPU time, not a marginal token fee. But exact per-request GPU accounting isn't feasible in the hackathon window, so we use a defensible estimate and say so in the invoice footnote. Regulated-enterprise honesty is on-brand: never quote a number you can't stand behind.

the approximation (10 §2)
# per local LLM call gpu_seconds ≈ wall_clock · (1 / avg_concurrent_reqs) # concurrency from vLLM /metrics # snapshot at dispatch & completion cost_usd = gpu_seconds · node_rate / 3600

Per scope = sum of its calls' estimates, cross-checked against node-level utilization samples; if the sum exceeds measured node time, scale down pro-rata. Two GPUs each doing 4 concurrent requests can't bill 8× a wall-clock second.

fallback when no concurrency snapshot
# estimate_gpu_seconds(backend, tokens_out) if backend in ("local","mock"): return max(0.1, tokens_out/40.0) return 0.0 # external/fireworks: per-token, no GPU line
! This tokens_out / 40 heuristic is the calibration knob — a real MI300X under real load reads differently. Real local dispatch overrides it with the measured value; the fallback only fires when the /metrics snapshot is missing.
§ Invoice footnote, verbatim: "GPU-seconds are an estimate (10 §2); token counts are exact." — the estimate is disclosed at the point of billing, not buried.

§4

The quote engine — deterministic, at stage 3

spec 10 §4 · quote/stage3.py build_quote_body()
0 intake 1 plan 2 certify 3 quote 4 build 5 consolidate 6 QA 7 deliver + invoice

The quote is a pure function of the certified plan's estimates + model_bom + rates — no LLM, fully reproducible. It emits an itemized range and the job parks at quoted until POST /approve-quote: the customer sees the price before a dollar of compute runs. Ranges = ±50% on token estimates, ±30% on GPU time.

how each line is computed
# inputs from the certified plan frontier_tokens, gpu_hours, fleet.width # frontier line uses the claude-fable-5 rate frontier_low = fT/1e6 · in_rate # all-input frontier_high = low + fT/1e6 · out_rate # +output # GPU hours split across stages, ×$2.00/GPU-h build_gpu = gpu_hours · 0.7 consolidate_gpu = gpu_hours · 0.1 qa_gpu = gpu_hours · 0.2 line_range = base·0.7 … base·1.3 # ±30%

The "projected per-run operating cost" line ([0.06, 0.12] per 100 units) is shown but excluded from the build total — it's opex, not capex. That distinction is the whole pitch (§6).

§5

The invoice — exact, aggregated at stage 7

spec 10 §5 · metering/invoice.py build_invoice()

At delivery (and on every refine), build_invoice(scope) sums the metered rows GROUP BY backend, maps each backend to its zone (registry ZONE), and emits one line per zone with real token counts and exact cost. It counts frontier_calls (EXTERNAL calls) and, if given the quote, computes delta_vs_quote. The result is stored as invoice.json in the immutable package and rendered in the Build view.

Refine invoice — the triage-authority receipt

10 §5 · refine/refine.py

When a change request comes in, the certifier triages it locally as amend (mechanical, scoped rebuild) or consult (structural, needs the frontier). The refine invoice adds one line the customer specifically wants to see:

Refine · triage = amend (stayed local)$0.00
frontier_consult_usd = 0.0   ← the receipt: "no data left the box for this change"
Refine · triage = consult (frontier)= total
frontier_consult_usd = invoice total   ← charged only when structure genuinely changed

§7

Budget guards — where spend gets stopped

spec 10 §7 · 07 §5.4 · router._budget_guard()

Guards run pre-dispatch, on the same seam that meters — before a call goes out, not after the bill lands. There is no path to spend that skips the check.

1

Fireworks daily cap

If fireworks_spend_today() ≥ $15/day (config fireworks_daily_budget_usd), emit system.notice (error) and raise BudgetExceeded. Caps the badged fallback so an outage can't run up an external bill.

2

Per-scope cap

Sandbox / per-run scopes register a cap via set_scope_cap(). If scope_spend(scope) ≥ cap → BudgetExceeded. Keeps a single judge run or sandbox from overrunning.

3

Live cost meter

meter.tick broadcasts scope totals ≤1/s; caps + current burn are exposed at /api/config/public for the UI's cost meter. Visibility, not enforcement.
