Seats & Two-Layer Routing

Application code never names a model. It addresses a seat — a role with a data-class clearance and a routing policy — and the router resolves that to a real backend at dispatch time. This page shows both routing layers: Layer 1 binds a seat to a backend/zone (the boundary is enforced here, in code); and Layer 2, inside a pooled seat, deterministically picks the pool member whose capability flags best fit the specific task. Plus BYOK connections and user-configurable seat rebinding. spec 02 §1–3, §7–8 · router.py · seats.yaml · models.yaml

01

Two routing layers, one principle

spec 02 §5 · §7 intro

The whole routing story is: code names roles, not models — at either layer. A dispatch (router.complete(seat, messages, data_class=…)) first resolves the seat to a concrete backend (Layer 1), then — only if that seat is a pool — chooses which pool member runs the task from its capability flags (Layer 2). Both layers are pure config: swapping a model, or changing which model handles SQL work, is a YAML edit, never a refactor.

LAYER 1 · seat → backend

Seat resolution

role → zone / data-class / binding · 02 §2, router.py
  1. seat → look up binding in seats.yaml
  2. sovereign overlay? swap to the sovereign binding (zero non-local calls, D7)
  3. backend health — fleet node down → fallback binding (Fireworks, badged)
  4. zone / data-class check → BoundaryViolation / SovereignViolation (fail closed)
  5. budget guard → BudgetExceeded, then dispatch
LAYER 2 · member selection

Capability argmax

within a pooled seat, best member per task · 02 §7.4, D12
  1. task carries task_flags (the planner declares them, 02 §7.3)
  2. candidates = healthy members of the pool
  3. score each member over the task flags (strong +2, ok +1, weak −2, absent 0)
  4. argmax → ties break to least-loaded, then round-robin
  5. deterministic — no LLM in the routing decision; recorded in task.started
i
Only coder (and process-mode per-unit steps) is a real multi-member pool, so Layer 2 fires only there. The other seven seats are single-binding — Layer 1 fully resolves them. Their capability flags still live in models.yaml so the catalog is complete. 02 §7.4 scope note

Why one seat goes external

The planner authors topology + bill-of-materials and benefits most from frontier reasoning — so its default is external claude-fable-5. It is allowed off-boundary only because its data-class ceiling is SANITIZED, enforced: router.complete raises BoundaryViolation the instant RAW is routed to a SANITIZED-ceiling seat, before any bytes leave. router.py L162 · 02 §1

Node B serves one glm-46 instance that backs certifier, conductor, consolidator, and the sovereign planner — different prompts, one set of weights. 03

Layer 2 — capability-aware member selection

spec 02 §7 · D12

Models are not interchangeable — one coder is strongest at greenfield generation, another at surgical refactoring, another at SQL, and a cheap Gemma is excellent at prose & extraction. A pool that round-robins throws that away. So each task in the build DAG carries task_flags from a small fixed vocabulary, and the scheduler runs a deterministic argmax over the pool's capability flags. No model gets picked by an LLM — the routing itself is arithmetic.

The selection algorithm

deterministic · no LLM
# runs in the worker scheduler (07 §5.3), per task
candidates = healthy members of the seat's pool
score(m) = Σ over task.flags:
    strong: +2   ok: +1   weak: −2   absent: 0
pick = argmax score
   ties      → least-loaded → round-robin
   no >0     → any healthy member + system.notice
   plan pin  → task{"model"} overrides (BoM-visible)

# recorded in task.started:
#   routing:{flags, chosen, score, considered}
i
The decision is written to build_tasks.assigned_backend and the meter events, so the BoM panel upgrades from "which model built which module" to "…and why". Unflagged tasks degrade gracefully to round-robin — v2.1 behavior. 02 §7.4

04

Worked example — one KYC/AML build, three tasks, three models

02 §7.3–7.4

A customer asks Nxcleus to build a KYC/AML onboarding process that screens applicants against the OFAC & EU sanctions lists. The planner (stage 1) decomposes it and, per the prompt contract, tags each module with task_flags. All three modules go to the same coder seat — but Layer 2 scores each task against the pool and routes three tasks to three different models, deterministically. Same weights every run.

A · Sanctions-screening engine — surgical edit into the existing matcher, wired to the fuzzy-match tool loop over a 262K-token rule corpus
refactor-editagentic-tool-uselong-context
qwen3-coder-next+2+2+2 = +6
devstral-small-2+2+2+1 = +5
qwen36-27b+2+1+1 = +4
gemma guest−2−2+1 = −3
→ qwen3-coder-next. Only member that is strong on all three: agent-tuned tool loops + a 262K window to hold the whole rule corpus while making a surgical change.
B · Risk-scoring data layer — schema + migrations + aggregation queries that roll sanction hits into an applicant risk score
sql-datagreenfield-codegen
qwen36-27b+2+2 = +4
qwen3-coder-next+1+2 = +3
devstral-small-2−2+1 = −1
gemma guest−2+1 = −1
→ qwen36-27b. The only pool member strong at sql-data — the routing exists precisely to catch this, where a round-robin might have handed schema work to devstral (weak).
C · Compliance runbook + audit trail — extract required fields from applicant PDFs and write the human-facing SAR/onboarding runbook
extractiondocs-writing
gemma guest+2+2 = +4
qwen36-27b+1+2 = +3
qwen3-coder-next−2+1 = −1
devstral-small-2−2−2 = −4
→ gemma-4-26b-a4b (guest). Best at extraction + prose, and it's already warm on node A — non-code work runs at zero extra serving cost instead of tying up a coder GPU.
✓
The WHY, not the what: the point isn't that three models ran — it's that the reason each ran is a recorded number, not a vibe. routing:{flags, chosen, score, considered} lands in task.started and on the invoice, so a compliance auditor can see the sanctions-matching code was built by the strongest available refactor+long-context model, and the SQL risk layer by the only SQL-strong one. Every one of these dispatches also stayed RAW inside LOCAL — real applicant names never left the walls. 02 §7.4 · D9
05

BYOK connections & user-configurable seats

spec 02 §8 · D13

The registry is extensible at runtime. Fable 5 is the planner's default, not a hardcode — every seat's binding is user-configurable, global or per-job. Customers (and the team in dev) can plug in their own API-backed models over the same capability-flag architecture, adding a fourth zone: custom.

01

Add API key

Write-only: encrypted at rest, surfaced masked, never serialized into events, logs, or packages.

02

Point at endpoint

Any OpenAI-compatible base_url. One connection carries many models.

03

Register models + flags

Each entry tagged with the same §7.2 vocabulary — capability routing treats it uniformly.

04

Rebind a seat

Global or per-job. Emits config.seat_bound, renders in the BoM — routing stays visible.

The CUSTOM zone in the boundary matrix

02 §8.3 · router.py _check_boundary
Connection stateSANITIZEDRAWSovereign
default · ceiling SANITIZED ✓ allowed ✗ blocked ✗ blocked
boundary-attested · ceiling RAW ✓ ⚠ customer-attested badge ✗ unless counts_as_local
attested + counts_as_local ✓ ⚠ badge ✓ eligible

A third-party API is outside the boundary by default (data_class_ceiling: SANITIZED). Raising it to RAW requires an explicit UI boundary attestation ("this endpoint is inside my trust boundary" — the customer's own on-prem vLLM / private-VPC). counts_as_local: true is the "on their metal" story — it makes the connection eligible for RAW seats and Sovereign Mode.

Rebinding is enforced, not advisory

02 §8.2

A model is offered for a seat only if its connection's zone/ceiling satisfies the seat's data class — the eligibility check runs before the option ever appears:
