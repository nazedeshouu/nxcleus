Backend AI & Routing · 03 / Pipeline & Orchestrator

The event-sourced engine

One custom asyncio engine drives every job through a nine-state machine. It dispatches, retries, and resumes — it never decides. The deciding lives in artifacts (the certified plan) and two bounded review points: the certifier before the build, and the conductor between the topological waves of stage 4. Every transition is an append-only event, so the same log gives you the live UI, replay, and crash-resume for free. This page traces that machine end to end and walks a two-wave KYC/AML build where the conductor amends wave 2 before green-flagging it.

Why it's built this way

Mind, hands, eyes — the engine never decides

The temptation is to hand orchestration to an LLM agent that loops and improvises. Nxcleus deliberately does not (D3): an improvising orchestrator can't be replayed, resumed after a crash, or audited. Instead intelligence is placed — the frontier planner authors a plan, a local certifier completes and certifies it, and a local conductor reviews between waves. The engine in the middle is boring on purpose. spec 07 §intro

✋

The hands — the engine

app/orchestrator/engine.py

A deterministic asyncio loop: while stage := next_stage(job). It dispatches each stage, retries on failure, parks on pause points, and resumes after a crash. It holds no opinions — it can't improvise, so it can always be replayed.

🧠

The mind — the certified plan

planner (external) → certifier (local)

The frontier planner designs the work topology from a sanitized brief; the local certifier completes it against RAW context, amends what sanitization blurred, and pins the goal. The plan is the decision, made once, up front.

👁

The eyes — the conductor

conductor (local, same GLM as certifier)

Between stage-4 waves it reads what was just built against the plan + goal and may issue bounded amendments to regions not yet built, or order rework of a finished module — then green-flags the next wave. D8

Pause points (⏸) are not threads sleeping — the engine parks the job (no task in memory) and an inbound API call re-submits it. That's why a deploy restart mid-quote costs nothing: there was nothing running to lose. spec 07 §2
Engine anatomy

One asyncio loop, no queues, no workers-as-processes

The whole engine lives inside the API process. No Celery, no Temporal, no IPC — SQLite has one writer and the fan-out is network-bound LLM calls, which asyncio handles at any concurrency this workload reaches. D3 spec 07 §1 Every stage receives the same StageContext — five capabilities, nothing else:

emit()append to events + fan out to SSE subscribers in-process
router.complete()dispatch to a seat → backend (02 §2), metered
daotyped DAO access to materialized tables
metertoken & GPU-second accounting per dispatch (10 §1)
checkpoint()job-scoped scratch — backs resumability
i
Because emit() writes the event and is the SSE source, there is exactly one path from "something happened" to both the database and the browser. The UI state for a job is literally a fold of its events; the DB rows are just a materialized cache for queries. spec 05 §1
Stage 4 · the heart of the engine

Topological waves & the conductor loop

Stage 4 is where D8 lives. The engine partitions the plan's DAG into topological waves (Kahn levels: wave 1 = tasks with no unbuilt deps, wave 2 = tasks unblocked by wave 1, …). Independent-parallelism plans collapse to a single wave; interdependent plans get the develop → review → develop rhythm. Within a wave, tasks dispatch in parallel to the coder pool; between waves the conductor reviews. spec 03 §6 spec 07 §3.1 Real loop below (fleet/stage4.py · seats/conductor.py):

Proceed-without-review is a hard rule. If the conductor call fails (2 router retries + 1 stage retry) or blows its 90 s timeout, the engine emits system.notice ("wave N unreviewed") and proceeds. A missing reviewer degrades quality, never availability. Scope-lock violations in an amendment are rejected and logged as plan.scope_violation, not applied — the same discipline as a constrained re-plan. spec 07 §3.1
Two ways a plan changes

Amendment vs consult — one stays home, one crosses the wall

Both edit the plan, but they are fundamentally different trust events. An amendment is a local seat patching the plan directly and immediately. A consult is an escalation back to the external planner — so it re-crosses the data boundary and must be sanitized first. The conductor has no consult lever; consults only exist at stage 2. spec 03 §5

▮ Amendment

A local patch to the plan. Never leaves LOCAL.
  • Who — certifier (stage 2) or conductor (between stage-4 waves). Both local, both RAW-access.
  • What — an RFC-6902 patch applied immediately; certifier patches anywhere, conductor patches only not-yet-built regions (scope-locked to the remaining DAG).
  • Audit — one amendments row, hash-chained sha256(prev_hash + patch + rationale), with origin. One chain, one seq, both origins. spec 05 §2
  • Boundary — local raw — no egress event.

◇ Consult

An escalation to the external planner. Re-crosses the wall.
  • Who — certifier only, at stage 2, when a finding needs the plan's author to re-think a region.
  • What — a scope-locked constrained re-plan request sent back to the planner (external); returned patch is rehydrated on application. Hard cap 3 rounds.
  • Gate — outbound payload re-masked against the boundary vault + swept by trust for residuals before dispatch. spec 03 §4.2
  • Boundary — external sanitized — appears in the egress ledger.
i
Both are best-effort by policy: a consult that fails against a real planner (out-of-scope re-plan, timeout, refusal) is caught, emits system.notice, and certification continues on the certifier's already-applied local amendments — mirroring the conductor's proceed-without-review rule. A bounded escalation must never hard-block the pipeline. spec 03 §4 (Wave-2 deviation)
Under stage 4

Worker pool, fleet manager & budget guards

Concurrency is not a queue — it's a semaphore. The coder SeatPool is built from the seat's bindings, the live node registry, and models.yaml capability flags. Each DAG-ready task is submitted with its plan task_flags; the pool picks the member by deterministic capability argmax (ties → least-loaded → round-robin; no flag match → round-robin — routing never blocks). The chosen backend lands in the task.started payload for the BoM panel. spec 07 §5.3 spec 02 §7.4

Fleet manager

local · MI300X

Nodes are cattle-by-registration: a booted droplet self-registers via POST /api/admin/nodes/register and whatever heartbeats is schedulable. The control plane polls node:/telemetry every 2 s; 3 missed beats → status=down → fleet.node_down → seats fail over to their fallback binding (badge on), the stage continues. spec 07 §5.1

If the BoM requests profile P3 but only P1 is registered, the engine emits fleet.profile_requested ("plan requests 3 nodes — provision or continue with fallback") and proceeds — demos never dead-end on missing GPUs. spec 07 §5.2

Budget guards — a pre-check on every dispatch

The router runs a guard before every LLM call; over-budget raises a 429-style refusal upward rather than spending. spec 07 §5.4 (real: models/router.py _budget_guard)

Per-run sandbox cap

Free from the event log

Replay & crash-resume, deliberately Temporal-free

Because the log is append-only and stages are idempotent, durability is almost nothing to build. On startup the engine scans for jobs.status NOT IN (done, aborted, quoted, blocked) and re-submits each — restarting from the top of the stage, not mid-stage. spec 07 §4 A restart during a live demo costs seconds, not the job.

01 · scan
Re-submit unfinished jobs

Lifespan hook lists jobs not in a terminal or parked state and spawns a _drive_job task for each.

02 · skip done work
Idempotent re-entry

Stage 4 skips build_tasks.status = done; run_units already written are skipped. A crash mid-build loses only in-flight modules.

03 · skip green-flagged waves
Waves remember

Wave membership persists on build_tasks.wave; resume re-enters at the first wave with incomplete tasks and skips conductor review for already green-flagged waves — the flag is an event, so replay knows.

i
Replay (the judge-facing feature) and resume are the same substrate read two ways: replay folds a completed job's events back into the UI for the gallery; resume folds an interrupted job's events to decide where to pick up. Amendments (conductor + certifier) live in one hash-chained sequence, so a replay can prove the plan's edit history was never rewritten. spec 05 §1–2
