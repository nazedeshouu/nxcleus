System Overview — one request, end to end

This is the map every other page zooms into. A single request enters at the left, walks the 0–7 pipeline spine, and for each stage you can read three things at a glance: which seat runs the work, which zone / host it runs on, and whether the meter ticks. Woven through it is the one claim the whole architecture exists to make good on — frontier-grade planning without a single byte of customer data leaving the customer's walls. Every token this platform generates is generated on AMD silicon. Of the eight seats, exactly one — the stage-1 planner — is designed to run off the customer's own GPUs, and it only ever receives a sanitized brief: structure without values, governed by the customer's own confidentiality policy. Every other seat runs local on self-hosted MI300X. Sovereign Mode rebinds even the planner on-fleet — zero non-local calls. (spec 01 §2, D7)

Host topology — three kinds of machine

spec 01 §1 · infra/fleet.yaml (profile P2)

An always-on control VM that runs zero inference (the receptionist, not the brain), one elastic 8× MI300X droplet partitioned into logical nodes A–D that runs all model work, plus two off-wall endpoints: Anthropic for the single planner call, and Fireworks purely to keep the URL alive between fleet sessions. The seal line down the middle is the data boundary — and the only thing that ever crosses it is a sanitized payload.

The 0–7 pipeline spine

spec 03 §1 · seats.yaml · router.py

One trip through the build pipeline is one job. Read each stage top-to-bottom: seat, zone, data class, and a meter dot that is lit whenever an LLM is dispatched (the router writes a meter_events row on every call — 10 §1) or dim when the stage is pure deterministic code. Notice the whole column is local except stage 1.

Intake · policy · boundary trust localnode A meter
Multi-turn elicitation builds a draft spec. trust distills the customer's confidentiality policy (doc / typed / voice-via-local-Whisper) into a structured RedactionPolicy, ingests codebase → code map and DB → schema-only, OCRs documents, then masks RAW → SANITIZED: PII + policy values become typed placeholders («PERSON_1»), raw→placeholder map stays in the local boundary_vault. Emits the planner brief.

The only seat that crosses the wall. The frontier planner (claude-fable-5) receives only the sanitized brief and is told exactly what fleet will execute its plan, so it designs for it: modules + typed interfaces + a DAG (or a corpus topology), each tagged with task_flags, plus a mandatory model BoM. Streams reasoning, emits Plan v1. Sovereign Mode rebinds this seat to local node B (GLM-4.6) — zero external calls.

The pass the frontier could never do (D9): a strong local model sees the full RAW context — original request verbatim, real files, real table/field names, boundary vault — runs a 7-check suite, applies hash-chained amendments, and rehydrates placeholders to real values. The certified plan becomes raw and never leaves local. Pins the goal. Any consult back to the planner re-crosses the wall through the sanitization gate (03 §4.2).

A pure function of the plan's BoM + estimates + rates.yaml — no model call, so the meter does not tick here. Emits an itemized quote range; the job pauses for approval. The invoice at stage 7 replays the real meter against this estimate.

4
Parallel build in waves coderconductor localnodes C·D + B meter
The DAG runs in topological waves. Tasks dispatch to the coder pool; the scheduler routes each task to the member best at its task_flags (SQL → qwen3.6-27b, refactor → devstral, greenfield → qwen3-coder-next). Between waves the conductor (same GLM-4.6 as the certifier) reviews outputs against the goal and may issue bounded amendments to not-yet-built regions before green-flagging the next wave (D8).

Consolidation consolidator localnode B meter
The brain's third prompt merges all module files into an assembled package (process.py implementing the runtime contract), then the full stage-2 integration suite runs in the --network none sandbox. Failures become defect tickets → coder fixes (≤3 rounds). Objective pass/fail, not review-by-vibes.

Adversarial QA + goal check inspectororaclecoder localnode A meter
A swarm of inspector agents probes the deployed-to-staging process over real HTTP; the oracle recomputes every numeric vector blind (self-consistency k=3). Disagreements file tickets → coder fixes → bounded loop. The final gate is the goal-fulfillment check: does what exists match what was originally asked, in the customer's own terms?

Where the boundary and the meter actually live

backend/app/models/router.py

The boundary is code, not policy, and the meter is not sprinkled through the stages — both live at one choke point. Every seat call in every stage above routes through router.py, which resolves the seat, applies the Sovereign overlay, then enforces the zone / data-class check before a single byte is dispatched, and writes the meter + egress row after. One gate = every token counted and every crossing audited.

The number that closes the demo. Because every crossing writes an egress_log row (ts, host, zone, seat, scope, bytes), a finished job produces a literal receipt: "exactly N external calls, all SANITIZED". Sovereign Mode makes N = 0, and any external egress during a sovereign job raises SovereignViolation and paints the run red — the network monitor reads this ledger, it is not cosmetic. (spec 01 §3)

Worked example — a KYC/AML onboarding request

trace through the spine

A bank cannot send applicant data to external AI — that is the entire reason this architecture exists. Follow one onboarding job through the spine and watch what the frontier is allowed to see versus what stays inside.

trust Stage 0 — intake & mask local
Inputs: applicant records (names, DOB, account numbers, national IDs), scanned ID documents, the applicants DB schema, and a policy doc — "never leak client legal names, account numbers, or our fee schedule." trust OCRs the docs locally, distills the policy into RedactionPolicy rules RP-1/RP-3, and masks: «PERSON_1», «ACCOUNT_A», table «TABLE_A»(id, «FIELD_1»:decimal). The raw→placeholder map lives only in boundary_vault. Emits boundary.sanitized: "7 PII fields masked, 3 docs OCR'd, RP-1/RP-3 applied".
planner Stage 1 — plan (crosses the wall) external
Fable 5 receives the sanitized brief: "KYC/AML customer onboarding," an Applicant entity with a PII-flagged full_name, a numeric rule "risk score = weighted sum…", and a mock sanctions_list connector. It designs modules mod_sanctions, mod_pep, mod_risk with typed interfaces and task_flags. Why this is safe: it never sees a real name, account number, or table — only structure. The egress ledger records one EXTERNAL / SANITIZED call.
certifier Stage 2 — complete & rehydrate local
Now with full RAW context, GLM-4.6 verifies the planner's sanitization-forced assumptions against reality (does «TABLE_A» actually have a unique applicant key? real column types?), amends gaps, and rehydrates placeholders to real names. Why re-introduce RAW: the frontier planned blind; the local pass fixes what sanitization blurred — legal because everything downstream is local (D7). Pins the goal: "Given an applicant record + documents, produce a risk-scored onboarding decision with sanctions/PEP screening and an audit trail."
coder conductor Stages 4–5 — build & merge local
mod_sanctions (flags sql-data · greenfield) routes to qwen3.6-27b; a refactor task would route to devstral. Wave 1 builds sanctions + PEP; the conductor reviews against the goal and amends mod_risk's scoring bands before wave 2 builds it. The consolidator assembles process.py and runs the full integration suite in the sandbox. Every token here is RAW-and-local.
inspector oracle Stage 6 — adversarial QA local
Inspectors hit the staged process with synthetic applicants; the oracle recomputes each risk score blind (k=3) and files a ticket on any disagreement. The goal-fulfillment check confirms the deliverable matches the bank's original ask.
trust Stage 7 — deliver local
The registered process ships with an invoice: "Frontier planning: 1 sanitized call, $2.40. Local certification + build + QA (GPU): $8.10." Every subsequent onboarding run executes with zero frontier calls — frontier intelligence was a one-time capital expense, not a per-record cost. See 04 · Economics & Metering.
