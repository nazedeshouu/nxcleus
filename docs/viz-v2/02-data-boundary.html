Backend Visualization · 02 — Data Boundary

The boundary is code, not policy.

Nxcleus runs a frontier planner it does not own, on hardware it does not control, over customer data that must never leave the building. The reconciliation is one mechanical rule enforced in the request path of every model call: data is typed RAW or SANITIZED, backends live in one of four zones, and a wrong pairing raises an exception before a byte is sent. This page traces exactly what may cross where, the two sanitization gates that produce the only thing the frontier ever sees, and the Sovereign switch that reduces off-box calls to zero.

Why "code, not policy"

The check runs on the dispatch path of every call

02 §4 · router.py

data_class is a required argument to the router. There is no code path to a backend that skips the zone check — it sits between seat resolution and dispatch, and it fails closed. This is the literal guard from backend/app/models/router.py (_check_boundary): 

# router.py — resolved backend r, data_class, sovereign flag
if sovereign and not (zone == "LOCAL"
        or (zone == "CUSTOM" and counts_as_local)):
    return None, "sovereign"   # → SovereignViolation

if data_class == "RAW":
    if zone == "LOCAL":        return ok
    if zone == "AMD_HOSTED":   return demo_badge  # flag-gated
    if zone == "CUSTOM" and ceiling == "RAW": return ok
    return None, "boundary"  # EXTERNAL+RAW = hard error

# SANITIZED travels everywhere; sovereign+EXTERNAL

How SANITIZED gets made

Two gates produce everything the frontier sees

03 §2.1 · §2.3 · §4.2 · D9 · D11

RAW does not become SANITIZED by convention — it passes through a gate that extracts values, substitutes typed placeholders, and sweeps for residuals. There are exactly two such gates: one outbound at stage 0 (composing the brief) and one outbound at stage 2 (consults going back to the planner). Between them, rehydration runs the substitution in reverse — but only ever inward, into the LOCAL zone.

RAW Stage 0 intake
Original request, files, code, DB schemas + the customer's confidentiality policy (doc / typed / voice→local Whisper).
  • trust distills a RedactionPolicy (D11)
  • PII baseline always on: names, accounts, gov IDs, secrets
  • policy adds company rules: product names, fee tables
→
🛡 Stage-0 sanitization gate
trust replaces every value with a typed placeholder «PERSON_1», «ACCOUNT_A». The raw→placeholder map stays in boundary_vault — never serialized outward.
  • emits boundary.sanitized + sensitivity report
  • cites the customer's own policy rule IDs
→
SANITIZED Planner brief
The SanitizedSpec — spec + context pack + policy summary, framed as a brief for a stronger model: strip values, keep structure.
  • the only artifact composed to cross to EXTERNAL
  • persisted to jobs.spec_json
2
Rehydration (D9) — the inward pass. At stage 2 the local certifier reads the full RAW context (request verbatim, raw code map, real schemas, the boundary vault) and resolves placeholders back to real values — «TABLE_A» → the real table name. The certified plan is thereafter data-class RAW, legal everywhere in LOCAL (which, per D7, is everything downstream) and illegal to serialize out. This is the whole point: fix what the frontier couldn't know, safely, because no seat after the planner ever talks to the cloud. 03 §4.1

🛡 The consult sanitization gate — the only RAW→out re-crossing

03 §4.2

When the certifier can't fix a plan defect locally, it consults the planner — which is EXTERNAL, so the request re-crosses the boundary. It cannot go out RAW. The gate does two things in order:

1 · Reverse-substitute

Mechanical re-masking of every known raw value against the boundary_vault — real names become placeholders again.

2 · Sweep for residuals

trust re-scans the payload for anything the mechanical pass missed, before dispatch.

3 · Rehydrate the reply

Returned patches arrive placeholder-based and are rehydrated on application, back inside LOCAL.

✓
The egress ledger shows each consult as EXTERNAL / SANITIZED — auditable in the demo. Consults are capped at 3 rounds; unresolved structural findings pause the job for human review (certify.blocked) rather than leaking to get an answer.
The switch

Standard vs. Sovereign Mode

D7 · 01 §3 · 02 §4

Sovereign Mode is a global toggle (also settable per job). It rebinds the one EXTERNAL seat — the planner — from Anthropic to the local node-B GLM, so the entire pipeline runs on the self-hosted fleet. It is not "less cloud"; it is zero non-local calls. Any EXTERNAL egress during a sovereign job raises SovereignViolation and paints the run red.

Worked example — KYC/AML onboarding

What the planner sees vs. what stays inside

03 §2.3 · §4.1 · D9 · D11

A bank asks Nxcleus to build a customer-onboarding screening flow. Their confidentiality policy says: never leak client legal names, account numbers, or our internal table names. Here is the same fact — an applicant record and its screening rule — as the frontier planner receives it, beside the RAW that never leaves the box. Same structure, everything identifying replaced.

The why · fidelity

The brief keeps enough structure — an Applicant with a PII full_name, a decimal risk field, a 3-factor rule — that a model which never sees the data can still design a correct sanctions/PEP/geo screening flow.

The why · safety

Not one real name, IBAN, table name, or the fee schedule crosses. The sensitivity report cites RP-1/RP-3 — the bank's own policy clauses — so sanitization is contractual, not discretionary (D11).

The why · correctness

Rehydration lets the local certifier bind «TABLE_A» to the real table and fix the planner's blind assumptions — legal because it happens after the last EXTERNAL call (D7/D9).
