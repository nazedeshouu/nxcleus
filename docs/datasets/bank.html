01 — Sandbox use cases

Three prompts, three planted mechanics

Scenario 1 of 3 — Dormant reactivation

Flag dormant accounts with unusual reactivation patterns and rank by risk

Flag dormant accounts with unusual reactivation patterns and rank by risk

Business value: surfaces the handful of accounts that suddenly move real money after years of silence — the classic precursor to a SAR filing — ranked so an investigator triages the highest-value few instead of hand-scanning all 107 dormant accounts.

How the pattern was planted

gen_bank() marks ~8% of the 1,200 accounts dormant (opened before mid-2021, last_active before June 2023). Of the 107 dormant accounts that results from that draw, the generator takes the first 15 dormant account IDs (dormant_ids[:15]) and inserts one extra transaction directly for each: a single large credit deposit of $40,000–$180,000 dated in June 2026 — the generator's own comment reads "sudden reactivation: big deposit in 2026 on a dormant account." The other 92 dormant accounts get no such transaction — true, uneventful dormancy — so a naive status='dormant' filter over-triggers; the model has to find the reactivation transaction, not just the status flag.

Evidence SQL

SELECT a.id, a.customer_id, a.status, a.last_active, a.balance_usd,
       t.ts AS reactivation_ts, t.amount_usd AS reactivation_amount
FROM accounts a JOIN transactions t ON t.account_id = a.id
WHERE a.status = 'dormant'
  AND t.direction = 'credit' AND t.kind = 'deposit'
  AND t.amount_usd >= 30000
  AND julianday(t.ts) - julianday(a.last_active) > 180
ORDER BY reactivation_amount DESC;

Scenario 2 of 3 — Structuring

Detect structuring-shaped deposit runs across transactions

Detect structuring-shaped deposit runs across transactions

Business value: finds transaction clusters shaped exactly like structuring — repeated deposits kept just under the $10,000 Currency Transaction Report threshold — the pattern BSA/AML programs are legally required to detect and file on.

How the pattern was planted

gen_bank() samples 12 accounts at random (r.sample(range(1,1201), 12)) and, for each, inserts 8–14 deposits of $9,000–$9,900 clustered between January and June 2026 — the generator's comment: "structuring: 8-14 deposits of $9,000-$9,900 within days." Ordinary deposits are drawn uniform(20, 8000), so they never reach $8,000 — the structuring band ($9,000–$9,900) sits in a gap no organic transaction can occupy, making it a clean signal once isolated by amount.

Evidence SQL

SELECT account_id, COUNT(*) AS deposit_count, ROUND(SUM(amount_usd),2) AS total_amount,
       MIN(ts) AS first_ts, MAX(ts) AS last_ts
FROM transactions
WHERE direction = 'credit' AND kind = 'deposit'
  AND amount_usd BETWEEN 9000 AND 9900
GROUP BY account_id
HAVING COUNT(*) >= 8
ORDER BY deposit_count DESC;

Scenario 3 of 3 — Sanctions-adjacent screening

Screen customers against the sanctions-adjacent name list

Screen customers against the sanctions-adjacent name list

Business value: screens the customer base against a list of names shaped close to real sanctions-list entries, so onboarding/KYC review catches a lookalike name before it becomes a live sanctions exposure.

How the pattern was planted

infra/seeds/_gen.py defines SANCTIONS_ADJACENT, six transliteration-style names close to real sanctions-list entries — the module comment is explicit: "deliberately shaped near real sanctions entries ... NOT real designated persons." gen_bank() assigns these to customer rows 1–6 in rotation. The first/last name pools that generate every other customer's name (_FIRST/_LAST in _gen.py) happen to contain the same given and family names ("Nadia", "Popova", "Ivan", "Petrov", "Andrei", "Marchenko" all appear in the general pools) — so six more customers coincidentally draw an identical full-name collision. A literal name IN (...) screen therefore returns 12 rows, not 6: a realistic texture of true intentional plants plus organic false-positive-shaped duplicates for the screening prompt to sort through.

Terms of Sensitive Data Use — Meridian Bank

Meridian Bank, N.A. ("Meridian", "the Bank") maintains this policy to govern how customer nonpublic personal information and Bank Secrecy Act records are classified, handled, and permitted to leave Bank infrastructure. It gives effect to the Bank's obligations under the Gramm-Leach-Bliley Act (GLBA) Safeguards Rule, the Bank Secrecy Act, and its anti-money-laundering (BSA/AML) program. Any automated or external processing of Bank data — including planning or analysis performed by a third-party or frontier model — is bound by these terms.

Data classification

The following fields in the Bank's customer and account systems are designated sensitive and carry the sensitivity class shown.

  • customers.national_id — customer government identifier (SSN / national ID). Class: RESTRICTED.
  • customers.name — customer legal name. Class: CONFIDENTIAL.
  • customers.dob — customer date of birth. Class: CONFIDENTIAL.
  • customers.email — customer contact email. Class: CONFIDENTIAL.
  • customers.phone — customer contact telephone number. Class: CONFIDENTIAL.
  • accounts.account_number — deposit account number. Class: RESTRICTED.
  • accounts.iban — international bank account number. Class: RESTRICTED.
  • accounts.balance_usd — account balance. Class: CONFIDENTIAL.
  • transactions.counterparty — transaction counterparty name. Class: CONFIDENTIAL.

Handling rules

  • Customer national identifiers in customers.national_id must never leave Meridian infrastructure in any form, masked or unmasked, and must never appear in materials sent to an external model.
  • Deposit account numbers in accounts.account_number and accounts.iban must be masked to the last four digits before any external transmission.
  • Customer names in customers.name may appear in external planning materials only in pseudonymized form; the mapping from pseudonym to real identity remains inside the Bank.
  • Customer contact details in customers.email and customers.phone must be masked before external transmission; an email domain may be retained but its local-part must not.
  • Account balances in accounts.balance_usd must be banded or aggregated before inclusion in any external planning brief; individual balances must not be transmitted.
  • Counterparty names in transactions.counterparty must be pseudonymized before external transmission, except where a name is screened against a lawful sanctions list.
  • All records governed by the Bank Secrecy Act, including customer identity and transaction detail, must remain resident on Bank-controlled infrastructure within the United States; cross-border transfer of BSA records is prohibited.
  • Suspicious-activity analysis — structuring and dormant-account reactivation review — must run on de-identified data only; customer identity is re-associated exclusively inside Bank systems.
  • Sensitive customer and account records must be retained for at least five years per BSA recordkeeping requirements and then securely destroyed.

This corpus is fully synthetic demo data; these terms exercise the platform's policy-distillation boundary.

06 — Further use & scaling

Regenerating and growing this dataset

Regenerate from scratch

Run the seed script from the backend package root. It rebuilds infra/seeds/out/bank.db and rewrites infra/seeds/out/terms/bank_terms.md in place.

uv run --project backend python scripts/seed.py bank

Determinism guarantee

Every draw comes from a local random.Random(SEED + salt) (infra/seeds/_gen.py, SEED = 20260709) — never the shared global random module. gen_bank() uses salt 1 for the primary stream (names, dates, planted episodes) and a dedicated salt 11 for GLBA/BSA-sensitive columns (SSN, IBAN, account number, email, phone) so a regenerate is byte-for-byte identical and adding new sensitive columns never perturbs the existing planted-pattern stream.

Row-count knobs

All in gen_bank(), infra/seeds/companies.py: the customer loop (range(1, 801), 800 rows), the account loop (range(1, 1201), 1,200 rows, ~8% dormant via r.random() < 0.08), and the base transaction loop (range(7600)). Planted-episode counts are separate literals: dormant_ids[:15] (reactivation accounts) and r.sample(range(1, 1201), 12) (structuring accounts, 8–14 deposits each). Scale any of these independently — the evidence queries key off amount/date shape, not row position.

Adding columns safely

Add new sensitive fields via a dedicated rng(salt) stream (as GLBA/BSA columns already do with salt 11) so existing planted patterns stay byte-identical across a regenerate. After any generator change, run the evidence check pattern from backend/tests/test_seeds_new.py — open the regenerated DB read-only, run each scenario's evidence SQL above, and assert the hit count still clears its floor (15 / 12 / 6 minimum for this company's three mechanics) before merging.

Prompt → sandbox UI mapping

The three prompts embedded verbatim in each scenario card above are the literal prompts array for id: "bank" in backend/app/api/sandbox.py's COMPANIES list — the judge sandbox renders them as one-click suggested prompts, and freeform prompts are also accepted (the planner's system prompt scopes topologies to this company's schema and refuses out-of-scope requests).
