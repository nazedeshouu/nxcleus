USE CASE 01

Intercompany reconciliation

Reconcile intercompany receivable and payable balances across all entity pairs and flag mismatches above materiality
Business value. Surfaces intercompany balances that won’t eliminate on consolidation before the close is locked.
How the pattern was planted. 60 of the 190 possible directed entity pairs carry an intercompany loan (A books account 1010 Intercompany Receivable, B books the mirror leg to 2010 Intercompany Payable). For 30 of those pairs, one transaction on the B-side payable is deliberately overstated by a random $5,000–$50,000 delta, so entity A’s receivable and entity B’s payable no longer agree — the other 30 pairs settle to the cent.
Evidence threshold: > $1,000
Evidence SQL 
SELECT a.entity_id AS entity_a, b.entity_id AS entity_b, a.recv AS receivable,
       b.pay AS payable, ROUND(a.recv - b.pay, 2) AS diff
FROM (SELECT entity_id, intercompany_entity_id,
             SUM(CASE WHEN side='dr' THEN amount ELSE -amount END) AS recv
      FROM gl_entries WHERE account_code='1010'
      GROUP BY entity_id, intercompany_entity_id) a
JOIN (SELECT entity_id, intercompany_entity_id,
             SUM(CASE WHEN side='cr' THEN amount ELSE -amount END) AS pay
      FROM gl_entries WHERE account_code='2010'
      GROUP BY entity_id, intercompany_entity_id) b
  ON a.entity_id = b.intercompany_entity_id AND a.intercompany_entity_id = b.entity_id
WHERE ABS(a.recv - b.pay) > 1000

USE CASE 02

Revenue misclassification

Find journal lines booked to expense-class accounts whose memos clearly describe sales revenue
Business value. Catches revenue mis-posted to expense accounts before it flows into a filing.
How the pattern was planted. 500 journal lines are written with a memo reading "Sales revenue – customer invoice #NNNNNN" but the credit leg lands on a random 5xxx expense-class account instead of a 4xxx revenue account — the narrative and the ledger classification disagree on 500 specific lines, spread evenly across all five years and all 20 subsidiaries.
Evidence SQL 
SELECT g.id, g.entity_id, g.account_code, a.name AS account_name,
       a.class AS account_class, g.amount, g.memo
FROM gl_entries g
JOIN chart_of_accounts a ON g.account_code = a.code
WHERE a.class = 'expense' AND g.memo LIKE '%Sales revenue%'

USE CASE 03

LEI remediation report

List counterparties with material ledger activity but a missing or malformed LEI
Business value. Builds the remediation worklist so materially-transacting counterparties carry a valid LEI.
How the pattern was planted. 80 of the 2,500 counterparties are planted with a NULL LEI or one whose length isn’t the ISO 17442 standard 20 characters (18/19/21/22-char strings). Every other counterparty gets a well-formed 20-character LEI. Each of the 80 gap counterparties is also given 3–6 dedicated high-value vendor payments, so the gap sits on real, material exposure rather than a dormant record.
Evidence threshold: > $100,000 total exposure
Evidence SQL 
SELECT cp.id, cp.name, cp.country, cp.lei, COUNT(*) AS gl_lines,
       ROUND(SUM(g.amount), 2) AS total_amount
FROM counterparties cp
JOIN gl_entries g ON g.counterparty_id = cp.id
WHERE cp.lei IS NULL OR LENGTH(cp.lei) <> 20
GROUP BY cp.id
HAVING SUM(g.amount) > 100000
ORDER BY total_amount DESC

USE CASE 04

Journal-batch integrity

Audit journal batches where total debits do not equal total credits
Business value. Finds out-of-balance batches that would break trial-balance integrity at close.
How the pattern was planted. 25 batches are built with 2–4 normal balanced dr/cr pairs, then one extra, unpaired debit line memo’d "Unposted accrual – review" is appended with no offsetting credit — every other batch in the ~325k-batch ledger nets to exactly zero by construction.
Evidence threshold: > $0.01 imbalance
Evidence SQL 
SELECT batch_id,
       ROUND(SUM(CASE WHEN side='dr' THEN amount ELSE 0 END), 2) AS total_dr,
       ROUND(SUM(CASE WHEN side='cr' THEN amount ELSE 0 END), 2) AS total_cr,
       ROUND(SUM(CASE WHEN side='dr' THEN amount ELSE -amount END), 2) AS imbalance
FROM gl_entries
GROUP BY batch_id
HAVING ABS(SUM(CASE WHEN side='dr' THEN amount ELSE 0 END)
         - SUM(CASE WHEN side='cr' THEN amount ELSE 0 END)) > 0.01

03 · Schema

Data model

Five tables: four reference tables feed one fact table. Reference data (entities, chart of accounts, counterparties) is static per run; gl_entries is the ~644k-row journal everything is measured against.

entities

20 rows · The 20 Aldgate subsidiaries — one row each, with its SOX attestation signatory.
  • idINTEGER PK
  • nameTEXT
  • countryTEXT
  • currencyTEXT
  • officer_nameTEXT
  • officer_emailTEXT

chart_of_accounts

400 rows · 400 accounts: 100 asset, 80 liability, 40 equity, 80 revenue, 100 expense.
  • idINTEGER PK
  • codeTEXT
  • nameTEXT
  • classTEXT

counterparties

2,500 rows · 2,500 external vendors/customers the group transacts with, with settlement + tax identifiers.
  • idINTEGER PK
  • nameTEXT
  • countryTEXT
  • leiTEXT
  • tax_idTEXT
  • vat_noTEXT
  • ibanTEXT
  • contact_emailTEXT

fx_rates

5,480 rows · Daily rate-to-USD for 5 non-USD currencies, 2024-01-01 through 2026-12-31.
  • idINTEGER PK
  • dateTEXT
  • currencyTEXT
  • rate_to_usdREAL

gl_entries

643,599 rows · Every double-entry journal line, 2022-01-01 through 2026-12-31 — the fact table.
  • idINTEGER PK
  • entity_idINTEGER
  • account_codeTEXT
  • batch_idTEXT
  • dateTEXT
  • sideTEXT
  • amountREAL
  • currencyTEXT
  • counterparty_idINTEGER
  • intercompany_entity_idINTEGER
  • memoTEXT
gl_entries.entity_id → entities.id  the booking subsidiary
gl_entries.intercompany_entity_id → entities.id  nullable · the counterparty subsidiary on an IC leg
gl_entries.account_code → chart_of_accounts.code  which ledger account the line hits
gl_entries.counterparty_id → counterparties.id  nullable · set on ~60% of operational lines
gl_entries.currency + date ↝ fx_rates.currency + date  logical join for USD translation, no FK constraint

Aldgate Holdings plc — Terms of Sensitive Data Use

Purpose

Aldgate Holdings plc operates the group general ledger and consolidation systems for its twenty subsidiaries. As a dual-listed group, Aldgate is subject to Sarbanes-Oxley section 302 and 404 attestation, market-abuse controls over material non-public financial information, and counterparty data-protection obligations. These Terms govern how the sensitive fields held in the group ledger may be processed, and in particular what may be distilled into any brief transmitted to an external planning model outside Aldgate infrastructure.

Data classification

The following fields carry a sensitivity class. External-transmission rules in the next section are keyed to these classes.

  • counterparties.tax_id — Restricted. Counterparty national tax identifier.
  • counterparties.iban — Restricted. Counterparty settlement bank account number.
  • counterparties.vat_no — Confidential. Counterparty VAT registration number.
  • counterparties.contact_email — Confidential. Counterparty finance-contact personal email.
  • counterparties.lei — Public. ISO 17442 Legal Entity Identifier.
  • counterparties.name, counterparties.country — Internal. Counterparty master reference.
  • entities.officer_name, entities.officer_email — Confidential. Subsidiary attestation signatory identity.
  • entities.name, entities.country, entities.currency — Internal. Subsidiary reference data.
  • gl_entries.memo — Confidential. Journal narrative that may embed counterparty identifiers.
  • gl_entries.amount — Restricted before filing. Material non-public financial data.
  • chart_of_accounts.code, chart_of_accounts.class — Internal. Ledger structure.

Handling rules

  • Counterparty national tax identifiers (counterparties.tax_id) must never leave Aldgate

    • infrastructure and must be fully masked before any brief is sent to an external planning model.

  • Counterparty bank account numbers (counterparties.iban) must be masked to the last four

    • characters before any external transmission.

  • Counterparty VAT registration numbers (counterparties.vat_no) may not appear in materials

    • transmitted outside the group finance perimeter.

  • Counterparty finance-contact email addresses (counterparties.contact_email) must be redacted

    • before external transmission and must never be paired externally with the counterparty legal name.

  • Subsidiary attestation officer identities (entities.officer_name, entities.officer_email)

    • may appear in external planning materials only in pseudonymized form.

  • Journal narrative memos (gl_entries.memo) frequently embed counterparty names, invoice numbers

    • and account details; any such identifier in a memo must be masked before external transmission.

  • Pre-close ledger amounts (gl_entries.amount) aggregated by subsidiary (gl_entries.entity_id)

    • constitute material non-public information and must not be transmitted outside Aldgate infrastructure before the corresponding results are publicly filed.

  • Legal Entity Identifiers (counterparties.lei) are public reference data and may be transmitted,

    • but must never be transmitted alongside the same counterparty's counterparties.tax_id or counterparties.iban.

  • Records for subsidiaries domiciled in the European Union (entities.country of DE or FR) and their

    • related counterparty data must be processed only within EU/EEA infrastructure.

  • Counterparty identifiers (counterparties.tax_id, counterparties.iban, counterparties.vat_no)

    • are retained no longer than seven years after the last related entry in gl_entries, then destroyed.

Closing note

This corpus is fully synthetic demo data; these terms exercise the platform's policy-distillation boundary.

Stage-0 wiring: these Terms are auto-distilled into the masking policy applied to every job run against this company — when a sandbox run names company: "ledger" and supplies no explicit policy, resolve_policy_sources() defaults the policy source to terms/ledger_terms.md, so Restricted/Confidential fields are masked before any brief leaves Aldgate infrastructure.

07 · Growing this dataset

Further use & scaling

Regenerate

uv run --project backend python scripts/seed.py ledger

Or run the module directly: python -m infra.seeds.ledger — its __main__ block regenerates and asserts every planted count before printing planted OK.

Determinism

Four independent random.Random(SEED + salt) streams (rng(12) ledger/entities, rng(1201) counterparties+LEI, rng(1202) FX, rng(1203) sensitive identifiers) — same seed, byte-identical output every run, and adding a new identifier stream never perturbs an existing one.

Row-count knobs

  • _ENTITIES list — 20 subsidiaries
  • ranges in the chart-of-accounts loop — 400 accounts
  • range(1, 2501) — 2,500 counterparties
  • range(320000) — the bulk operational ledger
  • r.sample(pairs, 60) / idx < 30 — IC pairs / mismatched half
  • range(500), r_cp.sample(range(1,2501), 80), range(25) — the other three planted counts

Adding columns safely

Draw new fields from a dedicated rng(<new salt>) stream (never reuse an existing one) so existing columns stay byte-identical. Re-run backend/tests/test_seeds_new.py after any change — the four evidence floors (30 / 500 / 80 / 25) must stay green.

Prompts → sandbox UI

The four prompts on COMPANIES in backend/app/api/sandbox.py (id "ledger") are what a judge sees and can click in /sandbox; each maps 1:1 to a scenario card above and triggers a real pipeline run via POST /sandbox/runs.
