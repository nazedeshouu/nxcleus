## One-liner

**"Describe an internal process. Get it running inside your walls — built, verified, and yours to run forever."** An adaptive platform for regulated enterprises: a plain-language description of an internal process becomes a running, verified, sovereign process automation — while customer data never leaves customer-controlled AMD hardware.

## The pitch (for the deck)

- **Problem:** regulated enterprises (banks, insurers, fintechs, healthcare, legal) can't hand their data or systems to external AI — so internal processes still run on consultants, manual work, and months-long software projects. A frontier chat can't do these jobs anyway: too large for one context, data can't leave the boundary, and trust requires verification against live systems.
- **Solution — a lifecycle, not a factory:**
  - **Build (once):** local trust layer (intake — prompt, files, codebase, DB schemas — plus the customer's **confidentiality policy**: upload it, type it, or dictate it to local Whisper; PII firewall + policy-governed masking; OCR) → frontier planner that only ever sees the sanitized **planner brief** and designs the *work topology* (independent or interdependent parallelism) + model bill-of-materials → local GLM **completes, rehydrates and certifies** the plan against the full raw context the frontier never saw (triage: small gaps patched locally with a signed amendment log; hard/structural issues escalated back as *sanitized* scoped consults) and pins the **goal** — the sentence the job will be judged against → elastic parallel fleet on AMD MI300X builds in **waves** (coder models with declared touch points, or corpus fan-out for processing jobs), a **conductor** reviewing between waves → local consolidation gated by objective tests → adversarial QA with an independent **Numeric Oracle** + a final **goal-fulfillment check** → the process lands in the **operations registry**.
  - **Operate (forever):** the certified process runs on new data fully locally — zero frontier calls, metered per run, oracle spot-checks as a live warranty.
  - **Refine (on demand):** change requests re-open planning under the same triage model; every refinement ships a versioned, diffed, re-certified v(n+1); old versions stay runnable.
- **Adaptive delivery modes:** the planner designs the best structure per task — **build mode** (construct a process automation), **process mode** (fan local models out over a corpus — documents, records, codebases — one unit per worker, aggregate into a dashboard/dataset), **semi-automated mode** (human-review steps designed in). All modes produce saved, re-runnable, refinable workflows.
- **Target user:** regulated enterprises. Wedge sentence: **"we operate where external AI is banned."**
- **Money slide:** **frontier intelligence is a capital expense, not a marginal cost** — paid once at plan time, on sanitized specs; every run after that is local and cheap. Competitors pay frontier tokens *and* surrender data on every run.
- **Boundary in one sentence:** the stage-1 planning conversation is the *only* designed-in external call in the entire lifecycle — it sees a sanitized brief governed by the customer's own policy; everything else runs on customer-controlled AMD hardware (or dedicated AMD capacity we operate), and **Sovereign Mode** moves even planning on-fleet: zero external calls.
- **Moat / originality:** the data-boundary architecture with customer-policy-governed sanitization; the triage authority model (frontier authors, local model finishes, rehydrates and certifies against data the frontier never saw, frontier on-call — with a visible amendment log); the goal anchor (delivery verified against the original ask, not the drifted plan); conductor-reviewed wave orchestration; **capability-aware model routing** (every model flagged by what it's best at — refactors to the editor, SQL to the data model, prose to Gemma — with the routing decision visible per module); adaptive topology planning with a model BoM driving GPU provisioning; dual-implementation QA via the oracle; Sovereign Mode toggle; lifecycle-as-asset.

## AMD compute plan (mandatory — load-bearing)

- [x] **AMD Developer Cloud GPUs (ROCm):** one elastic 8× MI300X droplet (1× for validation/idle) running vLLM — trust layer (stage 0), plan certification (stage 2), conductor + parallel fleet: coders or corpus fan-out (stage 4/4′), consolidation (stage 5), QA inspectors + oracle (stage 6), the Sovereign-Mode planner, **and all operate-phase runs** — per D7, every seat except the default frontier planner. Live ROCm telemetry in the Build view.
- [x] **Fireworks AI API (AMD-hosted):** the **resilience layer** — keeps the always-on live URL serving between fleet sessions and fails over if a node dies (clearly badged in the UI). Real integration (client, routing, P0 profile), but no designed-in seats: the product story is that nothing but the sanitized planning conversation ever leaves customer-controlled hardware (D7).
- [x] **Gemma:** two benchmark-justified roles, both merit-defensible (research pass 2026-07-09, catalog spec 11) — **Numeric Oracle** (gemma-4-31B: AIME 89.2, dense/stable, and deliberately *lineage-independent* from the Qwen coder fleet — the dual-implementation story) and the **trust layer** (gemma-4-26B-A4B: multimodal for OCR, 140-language, strong policy/prose work). The inspector seat moved to Qwen3.6-35B-A3B on a measured ~2× agentic-tool-use gap (MCPMark 37.0 vs 18.1) — honest numbers over prize-chasing; a mixed swarm with Gemma members is the optional third touchpoint. Gemma 4 is now Apache 2.0. Validation gate before demos; go/no-go on the $2,000 Gemma prize.
- [x] **Visibility to judges:** README architecture section + diagram, vLLM/ROCm/Fireworks code paths in repo, live GPU telemetry in the demo UI, "where each stage runs" deck slide, **GPU allocation justified by the plan's model BoM**.
